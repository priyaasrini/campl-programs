\section{Introduction}

Categorical Message Passing Language (CaMPL) is a functional-style concurrent programming language which implements concurrency by message passing. The type system of CaMPL originates from the Cockett and Pastro’s paper titled “The Logic of Message Passing”.The primary motivation of this paper was to define a concurrent analogue of the Curry-Howard-Lambek correspondence for functional programming \cite{CoP07}. 

Cockett and Pastro described a two-tiered logic consisting of a sequential part --- called the message logic --- that interacts with a 
concurrent part --- called the message-passing logic --- in which concurrent processes pass messages
of sequential data back and forth along communication channels. The categorical semantics of the message-passing logic is
given by a "linear actegory" in which the category of messages and sequential functions acts
on the category of communication channels and concurrent processes in two directions (sending messages towards the left or the right of the sender).

CaMPL programs are composed of multiple concurrent processes which communicate
with each other by passing messages along communication channels. At any instance in run-time, the topology of a CaMPL program is a finite acyclic graph consisting of processes which are nodes of the graph and channels which are the edges of the graph. The property of acyclicity at run-time is guaranteed by CaMPL's type system.
The current implementation of the CaMPL compiler performs both a type inference and type check at compile-time to ensure the compiled program follows the type system and will not deadlock or livelock. In this article, we introduce the features of CaMPL that constitute its concurrent type system and ensure that this guarantee can be made.

proc helloworld :: | Console => =

proc run =

\section{Processes}
\label{Sec:Processes}

Processes are the main actors of a CaMPL program. % i am not sure how i feel about ``main actors'' it's a bit vauge and informal, but i think it's fine.
A process is specified by the keyword \lstinline$proc$ followed by a process name, an optional type signature, and lists of the variables and channels used by the process.

In Example \ref{ex:helloworld}, the type signature of process \lstinline$helloworld$ was "\lstinline$ :: | Console => $" which indicates that it has access to one channel of type \lstinline$Console$.
This channel is given the name \lstinline$console$ as indicated by "\lstinline$ = | console => $" on the next line.
We will discuss type signatures in more detail in Section~\ref{Sec:channeltypes} on channel types.

A CaMPL program must have a process with the name \lstinline$run$ in which the execution of the program begins. This process is the main process and the only process which can create \tc{service channels}, such as the \lstinline$Console$, for communication with the outside world.

For now, we will consider the definition of process \lstinline$broadcast$ in Example \ref{ex:broadcastmsg}:
% priyaa suggested to make note of the observation that it would be great if we could define this recursively, but we can't without protocols because we need to be able to be able to change the type of the channel in the base case.
\begin{lstlisting}[label=ex:broadcastmsg, caption=Process that broadcasts a message from a single source] %can we change listing so that it says Example insted of Listing in the caption?

proc broadcast = 
	confirmation_msg | source, dest1 => dest2 -> do
		get msg on source								-- receive message from source
		put msg on dest1								-- send message to other processes
		put msg on dest2
		put confirmation_msg on source	-- send confirmation message to source
		close source									  -- close all channels and halt
		close dest1
		halt dest2  -- the last channel is closed with halt command
\end{lstlisting}
Observe that line 2 of the definition of \lstinline$broadcast$ consists of three comma-seperated lists of variables and channels used by the process: 1) variables, e.g. \lstinline$confirmation_msg$, 2) input polarity channels, e.g. \lstinline$source, dest1$, and 3) output polarity channels, e.g. \lstinline$dest2$.

The \tc{variables} are instances of sequential types and \tc{channels} are instances of concurrent types.
The channels on the left of \lstinline$ =>$ are called \tc{input polarity} channels. 
The channels on the right of \lstinline$ =>$ are called \tc{output polarity} channels. 
Lines 3 - 11 constitute the \tc{process body}.
We say that the channels listed in line 2 are ``in scope'' of the process body which means that the process can perform operations, called \tc{process commands}, on these channels.

The process body may be a single process command, as shown in the \lstinline$run$ processes in Examples~\ref{ex:helloworld} and~\ref{ex:serverecho}, or a \lstinline$do$ block of process commands, as shown in Example \ref{ex:broadcastmsg}. 
Process commands performed on the same channel can be grouped together with ``\lstinline$on ch do$'' as in Example~\ref{ex:serverecho} on line 3 of the client process.
This allows one to omit a repetitive ``\lstinline$on ch$'' after each command.






\section{Channels}
\label{Sec:Channels}

A process \tc{uses} a channel to interact with another process connected to the other end of the channel.
The type of a channel defines the interaction that the processes will have on it, and the polarity of the channel defines each process's role in the interaction. 
Thus, the process commands that a process can use on a channel depend on both the channel's type and polarity. 
Two processes can be \lstinline$plug$ged together if they both use a channel with the same type but opposite polarities. 
We will explain and justify the necessity of polarities in Section~\ref{Sec:polarities}.
Then, we will explain the interactions for each built-in channel type in Section~\ref{Sec:channeltypes}.
Finally, we will discuss custom channel types in Section~\ref{Sec:protocols}.

\subsection{Channel polarities}
\label{Sec:polarities}

Recall that processes are defined with two lists of {\em channels} that are in its scope: input polarity channels and output polarity 
channels. Different polarities refer to different ends of a channel. 

Recall Example~\ref{ex:serverecho} in which two processes are connected along a channel \lstinline$ch$.
Observe that \lstinline$client$ uses \lstinline$ch$ with output polarity (on the left end) and \lstinline$server$ uses \lstinline$ch$ with input polarity (on the right end).
%, i.e. the direction that messages will travel on the channel or the change in the topology of the process network that will occur.
In this case, the interaction is that messages will travel from left to right (\lstinline$client$ to \lstinline$server$) and then in the opposite direction from right to left.
%\comment{actually i think the discussion about types should not be here. we have not seen the Put or Get types yet.}
The polarity of the channel defines the process's role in the interaction. 
%To send or receive messages, the channel must be of type either \lstinline$Put$ or \lstinline$Get$.
%%, i.e. sending the message or receiving the message.
In this case, the process with the output polarity channel (on the left end) must use \lstinline$put$ and then \lstinline$get$, and the process with the input polarity channel (at the right end) must use \lstinline$get$ and then \lstinline$put$. 
%We will explain the \lstinline$Put$ and \lstinline$Get$ types in Section \ref{Sec:channeltypes}.

Defining the way a process uses a channel by both the type and polarity prevents ambiguity that can cause deadlocks.
For example, suppose we do not consider polarity and rewrite our code as follows:
\begin{lstlisting}[label=ex:ambiguityce, caption=Code without channel polarity (this code will not compile)] 
 proc run = 
 	| => -> plug				-- connects processes by shared channel ch
 		client( | ch )		-- client process has channel ch (no defined polarity)
 		server( | ch )		-- server process has channel ch (no defined polarity)
\end{lstlisting}

We know that we want \lstinline$ch$ to have a type indicating that a message should be sent and received on it.
%The processes know the type of the channel, but they can't definitively know whether they should be the one to send or receive the message.
However, if both processes are defined to first receive a message and then reply, they will both become stuck waiting for a message to be sent by the other:
\begin{lstlisting}
proc client = 
	| ch ->
		on ch do 
			get msg							-- receives server's message  
			put "Hello Server!"	-- sends message to the server      
			halt

proc server =  
	| ch ->
		on ch do
			get msg							-- receives client's message
			put "Hello Client!"	-- sends message to client
			halt
\end{lstlisting}
This is precisely what a deadlock is.
In this simple example, it is easy to see the mistake, but in more complicated programs, the type system with polarities has the ability to catch these sorts of errors.
Each channel type specifies the complementary roles of the processes on each end of the channel to ensure it is unambiguous.


\subsection{Built-in channel types}
\label{Sec:channeltypes}


The built-in channel types define the fundamental interactions that are permitted to occur between processes. 
The type of a channel can either be explicitly defined in the type signature of a process, or the compiler can infer the type of a channel based on how the processes on either end use it.

\subsubsection{\lstinline$Put$ and \lstinline$Get$ types}

As mentioned in Section~\ref{Sec:Processes}, processes can be defined with an optional type 
signature that indicates the types of the variables and channels it uses.
We will modify Example~\ref{ex:serverecho} and include the type signatures of the processes in the modified version.
We will define \lstinline$server$ to have a \lstinline$server_id$ variable that it will send back to the client instead of echoing the message:
\begin{lstlisting}[label=ex:serverid, caption=Server replies to Client with server id] 
proc client :: | => Put([Char]|Get(Int|TopBot)) = 			-- type signature
	| => ch ->
		on ch do 
			put "Hello Server!"	
			get int					
			halt

proc server :: Int | Put([Char]|Get(Int|TopBot)) => = 	-- type signature
	server_id | ch => ->				-- new server_id variable
		on ch do
			get msg							
			put server_id						-- sends server_id back
			halt
			
 proc run :: | => = 					-- type signature
 	| => -> plug					
 		client( | => ch )		
 		server( 1234 | ch => )		-- server process created with server_id
\end{lstlisting}
Type signatures have three comma-separated lists of sequential types, input concurrent types, and output concurrent types.
The names of the variables and channels are bound to the types in the order they appear.
We can observe that the type signature of \lstinline$run$ does not contain any types because it is not using any variables or any channels to interact with other processes.
%It creates the channel \lstinline$ch$ in \lstinline$plug$ command to connect the processes.

We can also observe that the type of \lstinline$ch$ is \lstinline$Put([Char]|  Get(Int| TopBot))$.
Recall that the type of the channel defines the interaction that the processes will have on it.
%, i.e. the direction that messages will travel on the channel or the change in the topology of the process network that will occur.
In this case, the interaction is that, first, a message of type \lstinline$[Char]$, i.e. a string, will travel from left to right (\lstinline$client$ to \lstinline$server$), second, a message of type  \lstinline$Int$ will travel from right to left (\lstinline$server$ to \lstinline$client$), and finally, the channel will be closed when the processes \lstinline$halt$.
From this we can see that a channel of type \lstinline$Put$ indicates messages will travel from left to right, type \lstinline$Get$ indicates messages will travel from right to left, and type \lstinline$TopBot$ indicates the interaction is over and the channel will be closed.

We can also notice that channel types are constructed inductively.
The outer-most layer is the first step of the interaction and the inner-most layer is the final step.
The base case is the \lstinline$TopBot$ type because it is always the final step to  \lstinline$close$ a channel or to \lstinline$halt$ by closing the final open channel.
The other types are inductive cases that each specify one step of the overall interaction.

\subsubsection{Tensor and Par types}

In our next example, we will consider a channel type that {\em bundles} two other channels together.
The bundled channel types \lstinline$(*)$, called \tc{tensor}, and \lstinline$(+)$, called \tc{par}, come from the mathematical underpinnings of CaMPL and allow changes to be made to the network of processes while ensuring no cycles are introduced.
By unbundling the channels, two new channels are created and passed into two new processes.

We will again modify Example~\ref{ex:serverecho} to demonstrate the bundling and unbundling of channels.
We will define another process \lstinline$two_clients$ that unbundles a channel and forks into two instances of \lstinline$client$ that both send a message to a modified \lstinline$server$ process:
\begin{lstlisting} [label=ex:clientfork, caption=Client forks to create two new processes] 
proc client :: | => Put([Char]|TopBot) = 		
	| => ch ->
		on ch do 
			put "Hello Server!"				  -- client sends string
			halt

proc two_clients :: | => Put([Char]|TopBot) (*) Put([Char]|TopBot) =
	| => two_ch ->
		fork two_ch as								-- unbundles channels ch1 and ch2
			ch1 -> client( | => ch1)		-- creates two new client processes
			ch2 -> client( | => ch2)		-- that each get one channel

proc server :: | Put([Char]|TopBot) (*) Put([Char]|TopBot)) => =  
	| two_ch => -> do
		split two_ch into ch1, ch2		-- unbundles channels ch1 and ch2
		on ch1 do											-- interacts with first client
			get msg						
			close
		on ch2 do											-- interacts with second client
			get msg						
			halt
			
proc run :: | => = 
 	| => -> plug						
 		two_clients( | => two_ch )		-- creates a single two_clients process
 		server( | two_ch => )				
\end{lstlisting}	
Observe that \lstinline$two_ch$ has type \lstinline$Put([Char]|TopBot) (*) Put([Char]|TopBot)$.
This is quite intimidating, but we have already discussed the \lstinline$Put([Char]|TopBot)$ type.
Let us use \lstinline$X$ as a shorthand for that channel type.
Then, we can consider the bundled channel type \lstinline$X(*)X$, pronounced ``{X \em tensor X}.''
The interaction this type defines is that the bundled channel will be replaced by two new channels, the process on the left (\lstinline$two_clients$) will be replaced by two new processes that each use one channel, and the process on the right (\lstinline$server$) will use both new channels to interact with each of the new processes according to type \lstinline$X$.
We could also write a program that has different interactions with the two new processes, and this would need a type that bundles two channels of different types like \lstinline$X(*)Y$.

We can see that a channel of type \lstinline$(*)$ indicates that one process on the left will be replaced by two new processes that are both connected to the process on the right.
What if we wanted the process on the right to be replaced by two new processes instead?
Then, we would use the other bundled channel type, \lstinline$(+)$, which is dual to the \lstinline$(*)$ type.



\subsection{Custom channel types}
\label{Sec:protocols}

We have covered the basic channel types, but the programs we have written so far notably lack the ability to send a variable number of messages.
Consider a modified version of \lstinline$broadcast$ from Example \ref{ex:broadcastmsg} in which the destination channels all have the same polarity and type:
% priyaa suggested to make note of the observation that it would be great if we could define this recursively, but we can't without protocols because we need to be able to be able to change the type of the channel in the base case.
\begin{lstlisting}[label=ex:modbroadcast, caption=Process that broadcasts a message on output channels (without confirmation)] 
proc broadcast :: | Put([Char]|TopBot) 
	=> Put([Char]|TopBot), Put([Char]|TopBot) = 
	 | source => dest1, dest2 -> do
		get msg on source								-- receive message from source
		put msg on dest1								-- broadcast message to other process
		put msg on dest2
		close source										-- close all channels and halt
		close dest1
		halt dest2    
\end{lstlisting}
This process is only able to broadcast a single message from the source.
To allow the process to call itself recursively until the source is finished sending messages, we need a way for a channel type to change from \lstinline$Put$ to \lstinline$TopBot$.
This can be achieved by defining custom channel types, called {\em protocols} and {\em coprotocols}, which can be recursively defined.

Let us consider the following example of a protocol called \lstinline$PassMessages$:  
\begin{lstlisting}[label=ex:protocolpassmsgs, caption=Protocol for passing an arbitrary number of messages] 
protocol PassMessages (A| ) => S =			-- passes messages of type A
    SendMsg :: Put(A|S) => S 						-- handle to send another message
    CloseCh :: TopBot => S 							-- handle to finish sending messages
\end{lstlisting}
This protocol allows an arbitrary number of messages of type \lstinline$A$ to be passed on a \lstinline$Put$ channel.
The two {\em handles}, \lstinline$SendMsg$ and \lstinline$CloseCh$, allow the type of the channel to be set to either \lstinline$Put$ or \lstinline$TopBot$.
In general, handles represent the set of valid interactions that may take place on a channel.
They can be thought of as session types.

%\comment{see if we can make this part more concise}
When we use \lstinline$PassMessages$ as a channel type, we will instantiate the sequential type variable \lstinline$A$.
In the case of  \lstinline$broadcast$, we will use the channel type \lstinline$PassMessages([Char]| )$ to pass strings.
%
This protocol does not use any concurrent type variables, but if it did, the list of the concurrent type variables would be on the right of the bar next to \lstinline$A$.
For example, if it took a concurrent type variable \lstinline$X$, we would have \lstinline$PassMessages(A| X)$.
%
The name of a protocol, its handles, and its sequential and concurrent type variables are required to begin with an uppercase alphabet character. 

Roughly speaking, protocols and coprotocols are the concurrent analogue of inductive and co-inductive data types in functional programming languages. 
The only difference between protocols and coprotocols is the direction that handles travel on the channel.
For a protocol, handles travel in the same direction as messages on a \lstinline$Put$ channel, and for coprotocols, handles travel in the same direction as a \lstinline$Get$ channel.
This means that a coprotocol is essentially a negated protocol, which is not true for sequential inductive and coinductive data types.

We can further modify \lstinline$broadcast$ to use the \lstinline$PassMessages$ protocol as follows:
\begin{lstlisting}[label=ex:recbroadcast, caption=Process that broadcasts messages from a single source (without confirmation)] 

proc broadcast :: | PassMessages([Char]| ) 
	=> PassMessages([Char]| ), PassMessages([Char]| ) = 
	 | source => dest1, dest2 -> do
		hcase source of					-- check whether there is another message
			SendMsg -> do
				get msg on source		-- receive message
				on dest1 do					-- broadcast message to other processes
					hput SendMsg			-- indicate there is another message
					put msg						-- send that message
				on dest2 do
					hput SendMsg
					put msg
				broadcast( | source => dest1, dest2) -- recurse
			CloseCh -> do					-- close all channels and halt
				close source							
				on dest1 do						
					hput CloseCh			-- indicate source is finished
					close							-- close channel
				on dest2 do
					hput CloseCh		
					halt	
\end{lstlisting}
Since protocol handles travel in the same direction as messages on a \lstinline$Put$ channel, \lstinline$broadcast$ is receiving handles on \lstinline$source$ and sending handles on its destination channels.
A handle is sent on a channel using \lstinline$hput$ which {\em activates} the channel and sets its type as specified by the handle.
A handle is received on a channel using \lstinline$hcase$ which allows a process to define a different process body for each possible channel type.

Originally,  \lstinline$dest1$ was an input polarity channel, so the process on the other end of the channel, which we will refer to as  \lstinline$proc1$, was using the output polarity end.
Now, \lstinline$proc1$ will need to use the input polarity end.
Suppose we do not want to change \lstinline$proc1$ to use the input polarity end.
Then, we can imagine that this output channel \lstinline$dest1$ connects \lstinline$broadcast$ to a \lstinline$send_to_dest1$ process that forwards messages to the final destination \lstinline$proc1$.
Coprotocols are the dual of protocols, so \lstinline$send_to_dest1$ can use a coprotocol version of \lstinline$PassMessages$ to forward the messages to \lstinline$proc1$.
We can define a coprotocol \lstinline$CoPassMessages$ as follows:
 \begin{lstlisting}
coprotocol S => CoPassMessages (A| ) =
    CoSendMsg ::  S => Get(A|S)
    CoCloseCh :: S => TopBot
\end{lstlisting}
Notice that the state variable \lstinline$S$ is on the left side of \lstinline$=>$ in the \lstinline$CoPassMessages$ definition instead of the right side in \lstinline$PassMessages$.
We need to use different handle names than we did in \lstinline$PassMessages$ because all handle names must be globally unique.

We can write \lstinline$send_to_dest1$ as follows:
\begin{lstlisting} [label=ex:coprotocoldest1, caption=Using a CoPassMessages channel] 
proc send_to_dest1 :: | PassMessages([Char]| ), CoPassMessages([Char]| ) => =
	| source, dest1 => -> 
		hcase source of					-- check whether there is another message
			SendMsg -> do
				get msg on source 		
				on dest1 do
					hput CoSendMsg		-- use coprotocol handle
					put msg
				send_to_dest1( | source, dest1 => ) -- recurse
			CloseCh -> do
				close source
				on dest1 do
					hput CoCloseCh
					halt
				
 \end{lstlisting}
 From this example, we can see that a coprotocol streamlines the process of writing code for a negated protocol rather than being a unique feature.

proc run =

As shown in this example, any number of processes can be connected using a single \lstinline$plug$ command as long as the process network remains acyclic.

Using only the programming features we have considered so far, a implementation for this \lstinline$server$ would need to interact with the \lstinline$client$ processes in a pre-determined order.
Thus, it would wait for the first \lstinline$client$'s message before beginning its interaction with the second \lstinline$client$.
Therefore, the second \lstinline$client$ would also wait for the first \lstinline$client$ in order to receive its echo from \lstinline$server$ and continue its execution.
This system can be described as parallel but not concurrent!

We want the order that \lstinline$server$ interacts with the \lstinline$client$s to be determined by which one it receives a message from first.
This controlled non-determinism uses the \lstinline$race$ process command.
We can modify \lstinline$server$ to hold a race as follows:
\begin{lstlisting} [label=ex:nondetserver, caption=Server echos each client in the order they send messages] 
proc server_deterministic = 
	| winner, loser => -> do
		on winner do		-- interacts with client it received a message from first
			get msg								
			put msg						
			close
		on loser do			-- interacts with other client
			get msg								
			put msg						
			halt
proc server = 
	| ch1, ch2 => -> race					-- controlled non-determinism via a race
		ch1 -> server_deterministic( | ch1, ch2 => )		-- client on ch1 wins
		ch2 -> server_deterministic( | ch2, ch1 => )		-- client on ch2 wins
\end{lstlisting}
Races can be held for any number of channels on which a process is waiting for a message.
In the above example, the \lstinline$race$ on line 12 consists of channels \lstinline$ch1$ and \lstinline$ch2$ as indicated by lines 13 and 14, respectively.
A process body is defined for each channel in the race, and the process will execute according to the winning channel's corresponding process body. 
In the above example, if \lstinline$ch1$ wins, line 13 will execute and \lstinline$server_deterministic$ will be called with \lstinline$ch1$ in the argument for the  \lstinline$winner$ channel.
If \lstinline$ch2$ wins, lines 14 will execute and \lstinline$server_deterministic$ will be called with \lstinline$ch2$ in the argument for the \lstinline$winner$ channel.
%Since only one of the race process bodies will be executed, all existing open channels in scope must be closed or passed into another process by the end of each process body.
%The \lstinline$race$ command must be the last command in a process block.

The type signature of both versions of \lstinline$server$ would be the same because the interaction on  \lstinline$two_ch$ is the same.
The only difference between the two versions is the order in which the steps of the interaction take place.


\section{ \textcolor{dkgreen}{PASS} Higher-order message passing}

In sequential functional programming, higher-order functions take other functions as input or produce other functions as output.
CaMPL's higher-order processes based on \comment{mathematical term} are a concurrent analogue of sequential higher order functions.

Higher-order processes encode other processes as sequential data, pass an encoded process as a message, and/or decode and invoke an encoded process.
Example \ref{ex:hoprocs} shows higher-order processes that pass an encoded \lstinline$helloworld$ process from Example~\ref{ex:helloworld}:
\begin{lstlisting} [label=ex:hoprocs, caption=Passing \lstinline{helloworld} between higher-order processes] 
proc ho_sender :: | => Put( Store(|Console=>) | TopBot) =
    | => ch -> do
        on ch do 
            put store(helloworld) 	-- encode helloworld and send as a message
            halt

proc ho_receiver :: | Put( Store(|Console=>) | TopBot), Console => =
    | ch, console => -> do
        on ch do
            get stored_process 			-- receive message with encoded helloworld
            close 
        on console do 
            hput ConsolePut 
            put ("Server says: Running the stored process")
        use(stored_process)( | console => )		-- decode and invoke helloworld
\end{lstlisting}
Observe that \lstinline$ho_sender$ and \lstinline$ho_receiver$ are connected along a \lstinline$Put(Store(|Console=>) | TopBot)$ typed channel.
The sequential type \lstinline$Store$ indicates that a message containing an encoded process is being passed.
The type signature of the encoded process, in this case \lstinline$helloworld$, is indicated within the \lstinline$Store$ type.

To encode a process, the built-in \lstinline$store$ function is used.
This function takes either the name of a previously defined process, as in the above example on line 4, or it takes an anonymous process as an in-line process definition.

To decode and invoke an encoded process, the \lstinline$use$ process command is used as in the above example on line 15.
To \lstinline$use$ an encoded process, a higher-order process must provide instances of variables and channels of the correct types and polarities as specified by the \lstinline$Store$ type signature.


\section{Discussion and future work}
%Races, Higher order message passing

The CaMPL project is still very much in progress.
On the implementation side, we are actively working on a paper that more comprehensively documents the details of the features that have been covered so far.
We are also working on updating the compiler to add the ability for processes to race protocol/coprotocol handles.
We would also like to make it possible to have a CaMPL program in which the processes are distributed over multiple devices that are connected by a network.

On the categorical semantics side, we are working on defining a precise semantics for non-determinism and considering how a non-deterministic semantics interacts with the other programming features.
We are also considering how a categorical semantics could be defined for message passing between quantum processes.
This would contribute to the development of programming languages for distributed quantum computing over a ``quantum internet.''
