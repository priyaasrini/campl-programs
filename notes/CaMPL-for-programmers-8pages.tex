\section{Introduction}

Categorical Message Passing Language (CaMPL) is a functional-style concurrent programming language which implements concurrency by message passing. The type system of CaMPL originates from the Cockett and Pastro’s paper titled “The Logic of Message Passing”.The primary motivation of this paper was to define a concurrent analogue of the Curry-Howard-Lambek correspondence for functional programming \cite{CoP07}. 

Cockett and Pastro described a two-tiered logic consisting of a sequential part --- called the message logic --- that interacts with a 
concurrent part --- called the message-passing logic --- in which concurrent processes pass messages
of sequential data back and forth along communication channels. The categorical semantics of the message-passing logic is
given by a "linear actegory" in which the category of messages and sequential functions acts
on the category of communication channels and concurrent processes in two directions (sending messages towards the left or the right of the sender).

CaMPL programs are composed of multiple concurrent processes which communicate
with each other by passing messages along communication channels. At any instance in run-time, the topology of a CaMPL program is a finite acyclic graph consisting of processes which are nodes of the graph and channels which are the edges of the graph. The property of acyclicity at run-time is guaranteed by CaMPL's type system.
The current implementation of the CaMPL compiler performs both a type inference and type check at compile-time to ensure the compiled program follows the type system and will not deadlock or livelock. In this article, we introduce the features of CaMPL that constitute its concurrent type system and ensure that this guarantee can be made.

\section{A brief overview of CaMPL}

As is tradition, we will introduce CaMPL with a ``Hello World!'' example program, show in Example \ref{ex:helloworld}.
We demonstrate a \tc{process} named \lstinline$helloworld$ printing \lstinline$"Hello World!"$ by sending the string as a message on a \tc{channel} named \lstinline$console$. 
The \lstinline$console$ channel is of type \lstinline$Console$ which is a special channel type built-in to the compiler to connect any CaMPL program to the terminal from which it is run.
\begin{lstlisting}[label=ex:helloworld, caption=Hello World!] %can we change listing so that it says Example insted of Listing in the caption?

proc helloworld :: | Console => = 
	| console => -> do
		hput ConsolePut on console
		put "Hello World!" on console    -- sends message to the console       
		hput ConsoleClose on console
		halt console                     -- closes console channel and halts               
\end{lstlisting}
The \lstinline$helloworld$ process can be invoked by calling it in the main process, \lstinline$run$, and giving it the \lstinline$console$ channel as follows:
\begin{lstlisting}
proc run = 
	| console => -> helloworld( | console => )  -- creates helloworld process
\end{lstlisting}

In Example \ref{ex:serverecho}, we will demonstrate communication between two user-defined processes: a client and a server. 
The client will send a string to the server, and the server will receive the string and echo it back.

\begin{lstlisting}
proc client = 
	| => ch ->

The client process has a (output) channel \lstinline$ch$ which connects it to the server.

proc server =

			halt
\end{lstlisting}	
The server process has a (input) channel \lstinline$ch$ which connects it to the client. 
It listens to the client and receives the client's message using \lstinline$get$, echos the message back using \lstinline$put$, and finally closes the channel and halts with \lstinline$halt$.

\section{Channels}
\label{Sec:Channels}

A process \tc{uses} a channel to interact with another process connected to the other end of the channel.
The type of a channel defines the interaction that the processes will have on it, and the polarity of the channel defines each process's role in the interaction. 
Thus, the process commands that a process can use on a channel depend on both the channel's type and polarity. 
Two processes can be \lstinline$plug$ged together if they both use a channel with the same type but opposite polarities. 
We will explain and justify the necessity of polarities in Section~\ref{Sec:polarities}.
Then, we will explain the interactions for each built-in channel type in Section~\ref{Sec:channeltypes}.
Finally, we will discuss custom channel types in Section~\ref{Sec:protocols}.

\subsection{Channel polarities}
\label{Sec:polarities}

Recall that processes are defined with two lists of {\em channels} that are in its scope: input polarity channels and output polarity 
channels. Different polarities refer to different ends of a channel. 

Recall Example~\ref{ex:serverecho} in which two processes are connected along a channel \lstinline$ch$.
Observe that \lstinline$client$ uses \lstinline$ch$ with output polarity (on the left end) and \lstinline$server$ uses \lstinline$ch$ with input polarity (on the right end).
%, i.e. the direction that messages will travel on the channel or the change in the topology of the process network that will occur.
In this case, the interaction is that messages will travel from left to right (\lstinline$client$ to \lstinline$server$) and then in the opposite direction from right to left.
%\comment{actually i think the discussion about types should not be here. we have not seen the Put or Get types yet.}
The polarity of the channel defines the process's role in the interaction. 
%To send or receive messages, the channel must be of type either \lstinline$Put$ or \lstinline$Get$.
%%, i.e. sending the message or receiving the message.
In this case, the process with the output polarity channel (on the left end) must use \lstinline$put$ and then \lstinline$get$, and the process with the input polarity channel (at the right end) must use \lstinline$get$ and then \lstinline$put$. 
%We will explain the \lstinline$Put$ and \lstinline$Get$ types in Section \ref{Sec:channeltypes}.

Defining the way a process uses a channel by both the type and polarity prevents ambiguity that can cause deadlocks.
For example, suppose we do not consider polarity and rewrite our code as follows:
\begin{lstlisting}[label=ex:ambiguityce, caption=Code without channel polarity (this code will not compile)] 
 proc run = 
 	| => -> plug				-- connects processes by shared channel ch
 		client( | ch )		-- client process has channel ch (no defined polarity)
 		server( | ch )		-- server process has channel ch (no defined polarity)
\end{lstlisting}

We know that we want \lstinline$ch$ to have a type indicating that a message should be sent and received on it.
%The processes know the type of the channel, but they can't definitively know whether they should be the one to send or receive the message.
However, if both processes are defined to first receive a message and then reply, they will both become stuck waiting for a message to be sent by the other:
\begin{lstlisting}
proc client = 
	| ch ->
		on ch do 
			get msg							-- receives server's message  
			put "Hello Server!"	-- sends message to the server      
			halt

proc server =  
	| ch ->
		on ch do
			get msg							-- receives client's message
			put "Hello Client!"	-- sends message to client
			halt
\end{lstlisting}
This is precisely what a deadlock is.
In this simple example, it is easy to see the mistake, but in more complicated programs, the type system with polarities has the ability to catch these sorts of errors.
Each channel type specifies the complementary roles of the processes on each end of the channel to ensure it is unambiguous.

\section{Controlled non-determinism}
\comment{We could potentially move this paragraph the code, and the paragraph below the code to fork-split.}
% priyaa suggested that we use the fork/split code and modify it for the discussion on races.
Recall Example~\ref{ex:clientfork} in which \lstinline$two_clients$ forks into two \lstinline$client$ processes and \lstinline$server$ splits the channel to interact with both new \lstinline$client$ processes.
Consider a modified version of \lstinline$server$ which echos the messages back to each \lstinline$client$:
%\begin{lstlisting} [label=ex:detserver, caption=Client forks to create two new processes] 
\begin{lstlisting}
proc server = | two_ch => -> do
		split two_ch into ch1, ch2		
		on ch1 do											
			get msg											-- waits for message from first client
			put msg											-- echos first client
			close
		on ch2 do											-- then interacts with second client
			get msg						
			put msg						
			halt			
\end{lstlisting}
Observe that \lstinline$server$ must wait for the message from the first client before it can begin its interaction with the second client.
This means that the second client is also waiting for the first client in order to receive its echo from \lstinline$server$ and continue its execution.
This system can be described as rather parallel but not concurrent!

To allow \lstinline$server$ to change the order in which it interacts with the two clients, we need \lstinline$server$ to be a non-deterministic process that dynamically adapts its execution at run-time depending on which client it receives a message from first.
This kind of controlled non-determinism can be achieved using the \lstinline$race$ process command.
We can modify \lstinline$server$ to hold a race as follows:
\begin{lstlisting} [label=ex:nondetserver, caption=Server echos each client in the order they send messages] 
proc server = | two_ch => -> do
		split two_ch into ch1, ch2		
		race												-- controlled non-determinism via a race
			ch1 -> do									-- first client wins
				on ch1 do								-- interacts with first client
					get msg								
					put msg						
					close
				on ch2 do								-- then interacts with second client
					get msg						
					put msg						
					halt		
			ch2 -> do									-- second client wins
				on ch2 do								-- interacts with second client
					get msg								
					put msg						
					close
				on ch1 do								-- then interacts with first client
					get msg						
					put msg						
					halt			
\end{lstlisting}
Races can be held for any number of channels on which the process is waiting for messages.
In the above example, the \lstinline$race$ on line 3 consists of channels \lstinline$ch1$ and \lstinline$ch2$ as indicated by lines 4 and 13, respectively.
A process body is defined for each channel in the race, and the process will execute according to the winning channel's corresponding process body. 
In the above example, the process body that will execute if \lstinline$ch1$ wins is defined on lines 4 - 12, and the process body for \lstinline$ch2$ is defined on lines 13 -21.
%Since only one of the race process bodies will be executed, all existing open channels in scope must be closed or passed into another process by the end of each process body.
%The \lstinline$race$ command must be the last command in a process block.

The type signature of both versions of \lstinline$server$ would be the same because the interaction on  \lstinline$two_ch$ is the same.
The only difference between the two versions is the order in which the steps of the interaction take place.


\section{ \textcolor{dkgreen}{PASS} Higher-order message passing}

In sequential functional programming, higher-order functions take other functions as input or produce other functions as output.
CaMPL's higher-order processes based on \comment{mathematical term} are a concurrent analogue of sequential higher order functions.

Higher-order processes encode other processes as sequential data, pass an encoded process as a message, and/or decode and invoke an encoded process.
Example \ref{ex:hoprocs} shows higher-order processes that pass an encoded \lstinline$helloworld$ process from Example~\ref{ex:helloworld}:
\begin{lstlisting} [label=ex:hoprocs, caption=Passing \lstinline{helloworld} between higher-order processes] 
proc ho_sender :: | => Put( Store(|Console=>) | TopBot) =
    | => ch -> do
        on ch do 
            put store(helloworld) 	-- encode helloworld and send as a message
            halt

proc ho_receiver :: | Put( Store(|Console=>) | TopBot), Console => =
    | ch, console => -> do
        on ch do
            get stored_process 			-- receive message with encoded helloworld
            close 
        on console do 
            hput ConsolePut 
            put ("Server says: Running the stored process")
        use(stored_process)( | console => )		-- decode and invoke helloworld
\end{lstlisting}
Observe that \lstinline$ho_sender$ and \lstinline$ho_receiver$ are connected along a \lstinline$Put(Store(|Console=>) | TopBot)$ typed channel.
The sequential type \lstinline$Store$ indicates that a message containing an encoded process is being passed.
The type signature of the encoded process, in this case \lstinline$helloworld$, is indicated within the \lstinline$Store$ type.

To encode a process, the built-in \lstinline$store$ function is used.
This function takes either the name of a previously defined process, as in the above example on line 4, or it takes an anonymous process as an in-line process definition.

To decode and invoke an encoded process, the \lstinline$use$ process command is used as in the above example on line 15.
To \lstinline$use$ an encoded process, a higher-order process must provide instances of variables and channels of the correct types and polarities as specified by the \lstinline$Store$ type signature.


\section{Discussion and future work}
%Races, Higher order message passing

The CaMPL project is still very much in progress.
On the implementation side, we are actively working on a paper that more comprehensively documents the details of the features that have been covered so far.
We are also working on updating the compiler to add the ability for processes to race protocol/coprotocol handles.
We would also like to make it possible to have a CaMPL program in which the processes are distributed over multiple devices that are connected by a network.

On the categorical semantics side, we are working on defining a precise semantics for non-determinism and considering how a non-deterministic semantics interacts with the other programming features.
We are also considering how a categorical semantics could be defined for message passing between quantum processes.
This would contribute to the development of programming languages for distributed quantum computing over a ``quantum internet.''
